{\heiti\zihao{4} 四、实验原理：}

SSL及其后续发展协议TLS是一种用于保障网络通信安全的协议，广泛应用于HTTPS中。SSL位于应用层与传输层之间，在不改变上层应用协议的基础上，为网络通信提供安全保护，其主要作用包括身份认证、数据保密性和数据完整性。

SSL协议主要由\textbf{记录协议}和\textbf{握手协议}两部分组成。记录协议负责对应用层传输的数据进行分段、加密和完整性校验，从而保证数据在传输过程中不被窃听和篡改；握手协议负责在通信开始前协商安全参数，包括协议版本、密码套件、密钥交换方式以及身份认证方式等。

SSL握手过程中，客户端首先向服务器发送 \texttt{ClientHello} 报文，声明自己支持的协议版本和加密算法；服务器返回 \texttt{ServerHello} 报文，并发送自己的数字证书。客户端验证服务器证书合法后，生成预主密钥，并利用服务器公钥加密后发送给服务器。随后双方根据协商结果生成会话密钥，并通过 \texttt{ChangeCipherSpec} 和 \texttt{Finished} 报文确认后续通信采用加密方式进行。握手完成后，应用数据通过SSL记录层进行安全传输。

SSL中的身份认证依赖于PKI和数字证书机制。数字证书通常采用X.509 v3格式，其中包含证书持有者信息、公钥、签发者、有效期及数字签名等内容。客户端在验证服务器证书时，需要检查证书是否在有效期内、证书主体名称是否正确、签发机构是否可信以及数字签名是否有效。只有验证通过后，客户端才会信任服务器身份并继续建立安全连接。

本实验中，需要完成根证书、Web服务器证书和客户端证书的申请、颁发与安装，并在IIS中配置SSL服务，使客户端能够通过HTTPS与服务器建立安全连接。结合Wireshark抓包分析SSL记录层和握手过程，可以进一步验证SSL与PKI在安全通信中的工作机制。

\textbf{（3）服务器证书申请、颁发和安装}

\textbf{(7)(8)Wireshark抓包，分析SSL记录与握手流程}

在客户端通过 \texttt{https://192.168.88.130} 与服务器建立连接的过程中，使用 Wireshark 对通信过程进行抓包分析。抓包结果显示，客户端与服务器首先完成 TCP 连接建立，随后进入 TLS 握手阶段。客户端向服务器发送 \texttt{Client Hello} 报文，服务器返回 \texttt{Server Hello}，之后双方继续传输 \texttt{Application Data}，说明 SSL/TLS 连接已经建立并进入加密数据传输阶段。由于本实验环境采用较新的系统和协议栈，抓包结果中 TLS 1.3 报文展示较传统教材中的 TLS 1.0/1.2 更加紧凑，但其核心流程仍然体现了客户端发起协商、服务器响应协商以及后续加密数据传输的完整过程。

进一步展开其中一条 TLS 报文，可以观察到 TLS 记录层的具体格式。实验中截取的报文为 \texttt{TLSv1.3 Record Layer: Handshake Protocol: Client Hello}。其中，\texttt{Content Type} 的值为 \texttt{Handshake (22)}，表示该记录承载的是握手协议数据；\texttt{Version} 字段显示为 \texttt{TLS 1.0 (0x0301)}，这是 TLS 1.3 中记录层用于兼容性保留的版本字段；\texttt{Length} 的值为 \texttt{258}，表示该记录层负载长度为 258 字节。继续展开 \texttt{Handshake Protocol: Client Hello} 可见，其中包含客户端支持的协议版本、随机数、会话 ID、密码套件以及扩展字段等内容，这些参数共同参与后续的 SSL/TLS 安全协商。

需要说明的是，由于本实验环境采用较新的系统和协议栈，Wireshark 中实际抓包结果显示为 TLS 1.3，部分握手报文在展示上较传统 TLS 1.0/1.2 更加紧凑，但其核心流程仍可概括为客户端发起协商、服务器响应协商、双方切换到加密状态并传输应用数据。
在 SSL/TLS 握手过程中，客户端首先发送 \texttt{Client Hello} 报文，向服务器声明支持的协议版本、随机数和密码套件；服务器返回 \texttt{Server Hello}，并发送服务器证书 \texttt{Certificate} 用于身份认证。客户端验证服务器证书后，发送 \texttt{Client Key Exchange} 完成密钥交换，随后双方通过 \texttt{Change Cipher Spec} 和 \texttt{Finished} 报文切换到加密状态。握手完成后，应用层数据以 \texttt{Application Data} 的形式进行加密传输。

\textbf{（9）分析证书}

通过对实验中生成和安装的 CA 证书、Web 服务器证书以及 Web 客户端证书进行查看，可以进一步分析三类证书在 SSL/TLS 体系中的作用。根证书用于建立信任锚点，服务器证书用于向客户端证明服务器身份，客户端证书则用于标识客户端身份。三类证书均采用 X.509 V3 格式，包含版本号、序列号、签名算法、颁发者、有效期、使用者、公钥信息等字段，这些属性共同决定了证书的用途与可信性。

从证书常规信息可以看出，客户端证书的“颁发给”为 \texttt{llb2023080904007}，其用途为“向远程计算机证明你的身份”，说明该证书用于客户端身份认证；服务器证书的“颁发给”为 \texttt{192.168.88.130}，用途为“保证远程计算机的身份”，说明该证书用于服务器身份认证；CA 根证书的“颁发给”和“颁发者”均为 \texttt{llb}，表明该证书为自签名根证书，用于建立本实验环境中的信任链。

{\heiti\zihao{4} 八、实验结论、心得体会和改进建议：}

本次实验围绕 SSL/TLS 与 PKI 的实际部署过程展开，完成了 IIS 服务、证书服务以及 HTTPS 安全访问环境的搭建，并对证书申请、证书颁发、证书安装、SSL/TLS 握手过程和记录层结构进行了综合分析。通过实验，成功配置了 AD CS 证书服务和 Web 注册功能，建立了实验环境中的 CA；完成了服务器证书和客户端证书的申请、颁发与安装，并在 IIS 中为默认网站成功绑定 HTTPS 服务；客户端能够通过 \texttt{https://192.168.88.130} 与服务器建立 SSL/TLS 安全连接。结合 Wireshark 抓包结果，可以观察到 \texttt{Client Hello}、\texttt{Server Hello} 以及后续 \texttt{Application Data} 等报文，验证了 SSL/TLS 在身份认证和加密传输中的实际作用。

通过对三类证书的查看与分析，可以进一步理解证书链在安全通信中的意义。CA 根证书用于建立信任锚点，服务器证书用于证明服务器身份，客户端证书用于证明客户端身份。实验中服务器证书与访问地址 \texttt{192.168.88.130} 保持一致，能够满足 HTTPS 场景中的身份校验要求；客户端证书则体现了基于 PKI 的身份标识功能。通过对证书版本、签名算法、有效期、颁发者、使用者、公钥信息等字段的分析，可以更加直观地理解 X.509 证书在 SSL/TLS 体系中的组织方式和安全属性。

在实验过程中，我对“证书申请—证书签发—证书安装—HTTPS 配置—握手建立—抓包分析”这一完整链路有了更加清晰的认识。相比只学习理论知识，亲自完成 IIS 配置、证书服务部署以及 TLS 报文分析后，更能够理解 HTTPS 建立安全连接的实际细节。例如，服务器证书的主体信息必须与访问标识匹配，否则容易在浏览器校验阶段出现问题；客户端证书能够体现用户身份；而 Wireshark 中记录层的 \texttt{Content Type}、\texttt{Version} 和 \texttt{Length} 等字段，则帮助我将抽象的协议过程和真实报文一一对应起来。虽然实验环境与教材示例在系统版本和 TLS 展示形式上存在差异，但这也让我进一步认识到不同系统和协议版本在实现细节上的变化。

本实验也暴露出一些可以改进的地方。首先，在实验指导中可以进一步区分不同 Windows Server 版本下 IIS 与证书服务界面的差异，减少因界面变化带来的理解困难。其次，对于客户端证书申请部分，可同时提供在线申请与 \texttt{certreq} 离线申请两种参考方法，以提高实验的适应性和成功率。再次，在抓包分析部分，可补充对 TLS 1.3 与 TLS 1.2 报文展示差异的说明，帮助实验者更准确地理解新版协议中更紧凑的握手结构。总体来看，本实验有效加深了我对 SSL/TLS 协议和 PKI 机制的理解，使我对网络安全中身份认证、信任链建立和加密通信的实现过程有了更加系统和直观的认识。
